# 物流儀表板資料模型與匯入規則 v0.1

本文先處理三件事：

1. 列出儀表板最終需要的 KPI。
2. 把每個 KPI 對應到資料來源。
3. 選「運費」當第一個閉環，定義匯入規則。

原則：先讓資料模型收斂，再讓畫面和程式跟著資料模型走。

## 3. 建議的標準資料模型

### 3.1 運費標準表 `freight_daily`

每一筆代表某一天、某一倉的運費。

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| date | string | 是 | 標準格式 `YYYY-MM-DD` |
| month | string | 是 | 標準格式 `YYYY-MM` |
| warehouse | string | 是 | 標準倉別：大溪倉 / 大肚倉 / 岡山倉 |
| freight_amount | number | 是 | 當日運費 |
| source_file | string | 是 | 來源檔名 |
| imported_at | string | 是 | 匯入時間 |

範例：

```js
{
  date: '2026-03-01',
  month: '2026-03',
  warehouse: '大溪倉',
  freight_amount: 12000,
  source_file: '每日運務費用.xlsx',
  imported_at: '2026-04-27 10:30:00'
}
```

### 3.2 工時標準表 `labor_daily`

每一筆代表某員工 / 派遣人員在某一天某作業區域的工時與費用。

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| date | string | 是 | 標準格式 `YYYY-MM-DD` |
| month | string | 是 | 標準格式 `YYYY-MM` |
| warehouse | string | 是 | 標準倉別 |
| vendor | string | 是 | 廠商 |
| shift | string | 是 | 日 / 中 / 夜 |
| employee_id | string | 是 | 員編 |
| department | string | 否 | 作業課別 |
| operation_area | string | 是 | 作業區域 |
| hours | number | 是 | 作業時數 |
| cost | number | 是 | 實際費用 |
| source_file | string | 是 | 來源檔名 |

### 3.3 揀次標準表 `picks_daily`

每一筆代表某一天、某倉、某作業區域的揀次。

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| date | string | 是 | 標準格式 `YYYY-MM-DD` |
| month | string | 是 | 標準格式 `YYYY-MM` |
| warehouse | string | 是 | 標準倉別 |
| business_type | string | 否 | 業務類別 |
| area | string | 否 | 作業區 |
| operation_area | string | 是 | 工時區域 / 作業區域 |
| picks | number | 是 | 揀次 |
| source_file | string | 是 | 來源檔名 |

### 3.4 預算表 `budget_monthly`

每一筆代表某月份、某倉、某費用類型的預算。

| 欄位 | 型別 | 必填 | 說明 |
|---|---|---|---|
| month | string | 是 | 標準格式 `YYYY-MM` |
| warehouse | string | 是 | 標準倉別 |
| cost_type | string | 是 | labor / freight |
| budget_amount | number | 是 | 月預算 |
| source_file | string | 是 | 來源檔名 |

### 3.5 對照表

| 對照表 | 目的 | 範例 |
|---|---|---|
| `warehouse_map` | 統一倉別名稱 | 大溪 / 大溪倉 / DC01 => 大溪倉 |
| `operation_area_map` | 統一工時與揀次的作業區域，由使用者維護 | EC理貨 / EC揀貨 => EC理貨 |
| `vendor_map` | 統一配送商或派遣商名稱 | 峻富物流 / 峻富 => 峻富 |
| `shift_map` | 統一班別 | 日 / 早 / 早班 => 日 |

## 4. 第一個閉環：運費匯入規則

### 4.1 閉環目標

先把「每日三倉運費」做穩。

成功標準：

1. 使用者上傳運費 Excel。
2. 系統解析出每日三倉運費。
3. 系統顯示預覽與合計。
4. 使用者確認後套用。
5. `運費損益分析` 的 F009 / F010 更新。
6. `總費用動支率` 的 T001 / T002 / T003 運費欄位更新。
7. 匯入狀態頁可以看到來源檔案、資料天數、三倉合計。

### 4.2 目前程式已支援的來源格式

| 項目 | 目前規則 |
|---|---|
| 必要工作表 | `進貨日與計價費用` |
| 日期欄位 | `列標籤` |
| 倉別欄位 | `大溪倉`、`大肚倉`、`岡山倉` |
| 忽略列 | 日期空白、包含 `總計` 的列 |
| 日期處理 | 目前從 `YYYY/MM/DD` 切出 `MM/DD` |
| 套用位置 1 | `DATA.freight.dailyByWarehouse` |
| 套用位置 2 | `DATA.dispatch.daily` 的運費欄位 |

### 4.2.1 已確認的運費明細來源

已檢查 `初始資料/運務費用.xlsx` 的 `貨運費用明細總表` 分頁。

| 項目 | 結果 |
|---|---|
| 分頁名稱 | `貨運費用明細總表` |
| 資料筆數 | 3324 筆，不含標題列 |
| 日期範圍 | 2026-03-01 ~ 2026-03-31 |
| 是否有負數 | 目前樣本無負數 |
| 整列完全重複 | 目前樣本無整列完全重複 |
| 可用於 F002-F005 | 是，分頁內有配送商、預計計價結果、到點計價結果、實際/調整後計價結果相關欄位 |

建議用途：

| 分頁 | 用途 |
|---|---|
| `進貨日與計價費用` | 每日三倉彙總，支援 F009、F010、T001、T002、T003 |
| `貨運費用明細總表` | 配送商與明細損益，支援 F002、F003、F004、F005 |

注意：`貨運費用明細總表` 是明細資料，因此同一天出現多筆資料是正常現象。D003 的「同一天但金額不同要擋下」只適用於每日三倉彙總分頁，不適用於明細分頁。

### 4.4 運費 Excel 必要欄位

| 欄位 | 必填 | 說明 |
|---|---|---|
| 列標籤 | 是 | 日期欄位，目前來源像 `2026/03/01` |
| 大溪倉 | 是 | 大溪倉每日運費 |
| 大肚倉 | 是 | 大肚倉每日運費 |
| 岡山倉 | 是 | 岡山倉每日運費 |

### 4.5 日期規則

接受格式：

| 原始格式 | 處理方式 |
|---|---|
| `2026/03/01` | 轉成 `2026-03-01` |
| `2026-03-01` | 保持 `2026-03-01` |
| Excel serial number | 轉成 `YYYY-MM-DD` |
| `03/01` | 不列為正式支援格式，因為已確認運費 Excel 日期一定有年份 |

已確認：

1. 運費 Excel 的日期一定有年份。
2. 正式日期格式以 `YYYY-MM-DD` 儲存。

### 4.6 倉別規則

標準倉別只允許：

```text
大溪倉
大肚倉
岡山倉
```

建議建立對照：

| 原始名稱 | 標準名稱 |
|---|---|
| 大溪 | 大溪倉 |
| 大溪倉 | 大溪倉 |
| 大肚 | 大肚倉 |
| 大肚倉 | 大肚倉 |
| 岡山 | 岡山倉 |
| 岡山倉 | 岡山倉 |

待確認：

1. Excel 是否可能出現其他倉別名稱？
2. 是否會有合計欄、北區/中南區欄、或其他不應匯入的欄？

### 4.7 金額規則

| 規則 | 處理 |
|---|---|
| 空白 | 視為 0 |
| 非數字 | 擋下匯入並提示欄位位置 |
| 負數 | 目前未確認是否可能發生；建議先列為警告，不直接擋下 |
| 三倉皆為 0 | 已確認要保留 |
| 小數 | 允許，但顯示時四捨五入為整數 |

待決策：

1. 運費是否可能有負數沖銷，目前未確認。
2. 若出現負數，建議第一版先顯示警告並允許套用，等確認業務規則後再決定是否改為擋下。

### 4.8 驗證規則

匯入前必須檢查：

| 驗證項目 | 不通過時 |
|---|---|
| 找得到工作表 `進貨日與計價費用` | 停止匯入 |
| 必要欄位都存在 | 停止匯入，列出缺少欄位 |
| 至少有一筆有效日期 | 停止匯入 |
| 日期可以轉成標準日期 | 停止匯入或列出錯誤列 |
| 金額可以轉成數字 | 停止匯入或列出錯誤列 |
| 日期沒有重複 | 目前需區分「完全重複列」與「同日多筆不同資料」 |
| 匯入資料期間與畫面日期區間一致 | 警告，不直接停止 |

待決策：

1. 如果是完全一樣的一列資料重複出現，建議視為重複匯出或貼上錯誤，預設擋下。
2. 如果是同一天但欄位資料不同，可能代表來源明細尚未彙總；若來源是樞紐彙總表，建議擋下；若未來允許明細匯入，才改成同日加總。
3. 匯入資料期間與畫面日期區間不同時，第一版先警告，不禁止。

### 4.9 預覽內容

使用者套用前要看到：

| 預覽項目 | 說明 |
|---|---|
| 來源檔名 | 確認上傳的是正確檔案 |
| 工作表名稱 | 確認解析來源 |
| 資料期間 | 最小日期 ~ 最大日期 |
| 資料天數 | 有效日期數 |
| 大溪倉合計 | 大溪倉運費合計 |
| 大肚倉合計 | 大肚倉運費合計 |
| 岡山倉合計 | 岡山倉運費合計 |
| 三倉總計 | 三倉合計 |
| 前 5 筆 / 後 5 筆 | 快速檢查資料是否偏移 |
| 警告訊息 | 例如空白、0、重複日期、月份不一致 |

### 4.10 套用規則

套用後更新兩個地方：

1. 運費分析資料

```js
DATA.freight.dailyByWarehouse = [
  ['03/01', 大溪倉運費, 岡山倉運費, 大肚倉運費]
]
```

2. 總費用動支資料

```js
DATA.dispatch.daily = [
  ['03/01', 大溪人力, 大溪運費, 大肚人力, 大肚運費, 岡山人力, 岡山運費]
]
```

注意：目前 `dailyByWarehouse` 的欄位順序是 `日期、大溪、岡山、大肚`，但預覽畫面是 `日期、大溪、大肚、岡山`。這不是錯，但容易讓後續維護者誤會，建議改成物件格式。

建議未來改為：

```js
{
  date: '2026-03-01',
  warehouse: '大溪倉',
  freight_amount: 12000
}
```

## 5. 建議開發順序

### 第 1 階段：運費閉環

只做這些：

1. 固定運費標準資料格式。
2. 加強運費 Excel 驗證。
3. 預覽時顯示期間、合計、警告。
4. 套用後更新 F009、F010、T001、T002、T003。
5. 保留匯入紀錄。

暫時不要做：

1. 配送商損益細節。
2. 人效 PPH。
3. 月度結算。
4. 後端資料庫。

### 第 2 階段：工時閉環

1. 固定 `labor_daily`。
2. 統一日期、倉別、班別、廠商、作業區域。
3. 驗證工時與費用。
4. 套用後更新 L001-L007 與 T001-T003 的人力欄位。

已確認匯入規則：

1. 正式來源為 `初始資料/全區工時.xlsx`，分頁優先讀取 `3月總`。
2. `倉別` 來源值 `大溪 / 大肚 / 岡山` 統一轉成 `大溪倉 / 大肚倉 / 岡山倉`。
3. `日期` 為 Excel 序號，匯入時轉成 `YYYY-MM-DD`。
4. `作業時數 = 0` 且 `實際費用 = 0` 的資料保留；KPI 計算工時與費用時自然不影響。
5. 筆數類指標不使用 row count，改用出勤人日或員編去重。
6. `檢核` 非 `ok` 的資料警告但允許套用。
7. 套用後更新 `T001 / T002 / T003` 的人力實際欄位。

年度預算匯入規則：

1. 正式來源為 `初始資料/年度預算.xlsx`。
2. 人力預算讀取分頁 `人力預算_轉換`，必要欄位為 `區域 / 月份 / 金額`。
3. 運費預算讀取分頁 `運費預算_轉換`，必要欄位為 `倉庫 / 類別 / 作業項目 / 月份 / 金額`。
4. 人力預算依 `區域 + 月份` 彙總，區域值轉成標準倉別。
5. 運費預算依 `倉庫 + 月份` 彙總，同一倉同一月的不同類別與作業項目加總。
6. 套用月份依畫面日期區間的月份判斷，例如 `2026-03-01` 對應 `3月`。
7. 套用後更新 `DATA.annualBudget` 與 `DATA.dispatch.budget`，並重新計算 `T001 / T002 / T003`。

### 第 3 階段：揀次閉環

1. 固定 `picks_daily`。
2. 統一倉別、作業區域。
3. 驗證揀次。
4. 套用後更新 P001-P005。

### 第 4 階段：人效

1. 用 `labor_daily` + `picks_daily` 計算 PPH。
2. 先只做倉別 + 作業區域層級。
3. 等對照表穩定後再做更細的課別或人員層級。

### 第 5 階段：月結

1. 用已驗證過的月資料。
2. 產出月度預算差異。
3. 產出主管 / 財務可讀的結算表。

### 6.3 名詞說明

「匯入月份」不是要求使用者一定要分月份上傳檔案，而是系統從 Excel 日期推導出的資料期間。例如檔案內日期是 `2026-03-01` 到 `2026-03-31`，系統就知道這批資料屬於 2026-03。

如果畫面目前篩選的是 2026-04，但使用者上傳的 Excel 內容是 2026-03，系統應提醒「畫面期間和匯入資料期間不同」。這是避免把 3 月資料誤套到 4 月視角，不代表檔案一定要每月切一份。
